EPIC-004 - Career Team Selector

Overview
The goal of this epic is to build the player team selector for Career Mode.

This component allows the user to choose their 7 starting players for the career:
- 1 Goalkeeper
- 6 Outfield Players

Players do not need positional assignments at this stage.
The user is simply choosing the 7 players they want to start their career with.

The selection flow is staged:
- Choice 1 is always a goalkeeper choice
- Choices 2 to 7 are always outfield player choices

Each choice presents exactly 3 randomly generated players.
The user must pick exactly 1 player from each set of 3 before progressing.

This means the full generated pool for the flow is:
- 3 goalkeepers for the goalkeeper choice
- 18 outfield players total for the 6 outfield choices
- The 18 outfield players are then distributed randomly into 6 groups of 3

The selection is complete and valid only when:
- 1 goalkeeper has been chosen
- 6 outfield players have been chosen

This epic should use the player generation system already created in EPIC-003.

================================================================
Core Behaviour
================================================================

Goalkeeper Choice
- The first selection screen/step should present 3 goalkeeper players.
- The user selects 1 of them.
- This completes the goalkeeper selection.

Outfield Choices
- The next 6 selection steps should each present 3 outfield players.
- The user selects 1 player from each set.
- This results in 6 outfield players selected overall.

Chosen Players
- A visible chosen-player list should always be displayed while making selections.
- The chosen list should show all selected players so far and their details.

Player Details
- While making each choice, all visible players should show their stats.
- Chosen players should also show their stats/details in the selected list.

Validation
- Team selection is valid only when the user has selected:
  - 1 goalkeeper
  - 6 outfield players

================================================================
Outfield Player Pool Rules
================================================================

The 18 outfield players must all be generated at once before being split into the 6 choices.

Target overall range for all 18 outfield players:
- minimum overall: 20
- maximum overall: 25

Desired frequency weighting across the 18-player pool:
- 20 = rare
- 21 = common
- 22 = slightly more common
- 23 = common
- 24 = uncommon
- 25 = rare

The system does not need to force a perfectly fixed count for each rating every time, but it should bias generation so the spread tends to follow that pattern.

================================================================
Forced Influence Rules in Outfield Pool
================================================================

To guarantee a good spread of player types, 10 of the 18 generated outfield players must be created using forced influence rules.

The following influence rules must each be forced exactly twice:
- Influence Defender
- Influence Defensive Mid
- Influence Attacking Mid
- Influence Attacker
- Influence Goalscorer

This creates 10 forced-influence players total.

The remaining 8 outfield players:
- should use the normal random influence rule behaviour from EPIC-003
- meaning the standard 1 in 6 rule selection should apply to those players

After all 18 outfield players are generated:
- shuffle them
- split them into 6 groups of 3
- use those 6 groups as the 6 outfield selection choices

FEAT-414 - Add Name, Appearance, and Potential to Generated Players
----------------------------------------------------------------
Status: done

Description
Expand the player generation output so every generated player includes a name, an appearance array, and a potential value in addition to their skill set.

Requirements
- Update the shared player generation system so every generated player includes:
  - name
  - appearance
  - potential
- These values must be generated as part of the standard player creation flow for both goalkeepers and outfield players.
- These values should be returned anywhere the player generator is used, including the team selector flow.

Name
- Generate a random name for each player.
- Build names from:
  - a pool of first names
  - a pool of last names
- Combine one first name and one last name to produce the final player name.
- The names should have a slight parody feel to them, but still feel believable enough for the football game world.
- Avoid using exact famous real player names where possible.
- Keep the name generator reusable and easy to expand later.

Suggested output shape for name
- Either:
  - name
- Or, if useful for future expansion:
  - firstName
  - lastName
  - name
- If split values are stored, name should still be available as the combined display value.

Appearance
- Generate an appearance array for each player.
- The appearance value must be an array of exactly 3 numbers.
- Each number must be a random integer from 1 to 5 inclusive.
- Example valid values:
  - [1,5,3]
  - [4,2,4]
  - [2,1,5]
- This value is for future use and does not need visual rendering in this feature.
- Keep it stored on the player object for later systems.

Potential
- Generate a potential value for each player.
- Potential is based on the player's generated current overall rating.
- Use this exact logic:

  Potential Max = 50 - Current Overall Rating

  Potential = random integer between 1 and Potential Max

- Use the player's final generated current overall, not the original target before generation corrections.
- Potential must always be valid within that calculated range.
- Because current overall is capped at 50 in this system:
  - lower-rated players can have larger potential ranges
  - players already near 50 have very limited room to improve
- Handle the edge case where Current Overall is 50 so the result remains valid and stable.
- The implementation should avoid producing invalid ranges or runtime errors when Potential Max is very small.

Implementation notes
- Do not bolt these values on in the UI layer.
- They should be part of the core generated player object returned by the shared generator.
- Keep each generation concern modular:
  - name generation utility
  - appearance generation utility
  - potential generation utility
- The generator should remain deterministic in structure and easy to test.
- Make sure these values flow through to any existing hooks/utilities that return generated players.

Acceptance Criteria
- Every generated player includes a random name.
- Names are built from a first-name pool and a last-name pool.
- Names have a slight parody feel without being direct copies of famous real players.
- Every generated player includes an appearance array of exactly 3 integers.
- Each appearance value is always between 1 and 5 inclusive.
- Every generated player includes a potential value.
- Potential is calculated from:
  - Potential Max = 50 - Current Overall
  - Potential = random integer between 1 and Potential Max
- The implementation handles high-overall edge cases safely.
- These new values are part of the shared generated player object and are available everywhere the generator is used.
