EPIC-009 - Manager Generation and Team Assignment

Overview
The goal of this epic is to generate managers for every non-player team in the career and assign each manager to a team at the start of the career.

Managers should be procedurally generated alongside the rest of the career data and saved into the game state as part of career generation.

For now:
- every other team in the game needs a manager
- managers stay fixed to their assigned team for the whole career
- manager movement between teams is out of scope
- manager preferences are not yet used to actively run matches, but they must be generated and stored now so later systems can use them

This epic is about:
- manager data model
- procedural manager generation
- manager-to-team assignment
- persistence in career state
- exposing the data for debugging

This epic is not about:
- manager movement between teams
- manager hiring/firing
- manager contracts
- manager names or portraits unless already easy to support
- tactical AI behaviour in matches
- manager history across seasons

================================================================
Core Manager Purpose
================================================================

Managers exist so that AI teams have tactical preferences.

These preferences will later be used in the match system to determine the opposition tactics used by that team in a match.

For this epic, we only need to:
- generate the managers
- assign them to teams
- store their tactical preference values
- make them available in career data for later use

================================================================
Which Teams Need Managers
================================================================

Every other team in the game needs a manager.

For this epic, that means:
- all generated AI teams need a manager
- the user-controlled team does not need an AI manager for this feature unless the existing model already benefits from having one as a placeholder

In practice this should include:
- League 5 AI teams
- League 4 teams
- League 3 teams
- League 2 teams
- League 1 teams
- Foreign teams / Champions Cup teams

Managers should be created during career generation and assigned immediately to their team.

================================================================
Manager Data Model
================================================================

Each manager holds the following values:

- preferredDefensiveTactics
- unpreferredDefensiveTactics
- preferredAttackingTactics
- unpreferredAttackingTactics

Defensive tactic pool:
- Low Block
- Mid Block
- High Press
- Offside Trap
- Zonal

Attacking tactic pool:
- Posession
- Long Ball
- Wing Play
- Counter
- Direct

Rules for manager preferences:

Preferred Defensive Tactics
- choose 2 or 3 defensive tactics

Unpreferred Defensive Tactics
- choose 1 or 2 defensive tactics

Preferred Attacking Tactics
- choose 2 or 3 attacking tactics

Unpreferred Attacking Tactics
- choose 1 or 2 attacking tactics

Important logic constraints:
- preferred and unpreferred selections for the same tactic type must not overlap
- a tactic cannot appear in both preferred and unpreferred for the same manager
- a manager does not need to explicitly rate every tactic, only store these chosen preference groups

================================================================
Generation Rules
================================================================

Manager generation should be procedural and randomised.

For each generated manager:
- choose 2 or 3 preferred defensive tactics
- choose 1 or 2 unpreferred defensive tactics
- choose 2 or 3 preferred attacking tactics
- choose 1 or 2 unpreferred attacking tactics

Selection rules:
- choices should come from the valid tactic lists only
- selections should be unique within each list
- preferred and unpreferred sets must not overlap within the same tactic category

The system does not need to simulate personality or style beyond these preference lists yet, but the output should be robust enough to support that later.

================================================================
Assignment Rules
================================================================

Each generated AI team gets exactly one manager.

For now:
- the manager is permanently assigned to that team for the career
- no reassignment or movement logic is needed
- no managerless AI teams should exist after career generation completes

The manager should be saved as part of the team data or linked team data structure in a way that is easy for future match logic to consume.

================================================================
Career Generation Integration
================================================================

This epic should integrate with the career generation flow built in earlier epics.

Manager generation should happen:
- during career creation
- after or alongside AI team generation
- before the career is considered fully generated and saved

The generated managers and their assignments should be stored in gameContext as part of the saved career state.

================================================================
Future Usage Context
================================================================

This epic does not yet need to implement tactical decision logic in matches.

However, the manager preference data must be structured so that later systems can use it to decide:
- which defensive tactic an AI team is likely to choose
- which attacking tactic an AI team is likely to choose

This means the stored format should be clean, explicit, and easy to consume later.
