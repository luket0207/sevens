EPIC-013 - Player Traits

Overview
The goal of this epic is to introduce player traits into the game.

Player traits do not alter:
- player overall rating
- tactic calculations
- DTR
- ATR
- tactic compatibility

Their purpose is to support other systems later, such as:
- player growth
- match chance behaviour
- other player-specific gameplay effects

For this epic, we do not need to implement what traits do.
We only need to:
- generate and assign them correctly
- store them on players
- display them in the player skillset component

Traits must be assigned when a player is generated.
This applies to:
- the user’s players
- AI team players
- goalkeepers
- outfield players

Trait assignment should be weighted by the generation influence rule used for that player.
Goalkeepers use their own goalkeeper rarity column.

================================================================
Core Trait Rules
================================================================

Traits are assigned during player generation.

Each player can receive:
- 0 traits
- 1 trait
- 2 traits

Use this exact distribution:

- 0 traits = 60%
- 1 trait = 32%
- 2 traits = 8%

If a player is assigned traits:
- use the player’s generation influence context
- use the trait rarity weights for that context
- randomly select up to the required number of traits
- a player cannot receive the same trait twice

Important scope rule:
- for now, traits are data only
- do not implement gameplay effects for the traits yet

================================================================
Trait Weighting Context
================================================================

Trait weighting depends on how the player was generated.

Use one of these contexts:

- Goalkeeper
- Influence Defender
- Influence Defensive Midfielder
- Influence Attacking Midfielder
- Influence Attacker
- Influence Goalscorer
- No Influence

Rules:
- If the player is a goalkeeper, always use the Goalkeeper rarity column.
- Otherwise, use the rarity column that matches the player’s applied influence rule.
- For outfield players with no influence, use the No Influence rarity column.

================================================================
Trait Display Rules
================================================================

Traits should be displayed inside the player skillset component.

This is intentional because:
- we always want to see traits at the same time as skillsets
- traits are part of the player identity display

Requirements:
- add a dedicated traits section inside the existing player skillset component
- the trait section should clearly list the player’s assigned traits
- players with no traits should still be handled cleanly
- the display should work anywhere the player skillset component is already used

For this epic, a simple list of trait names is enough.
If useful, trait IDs may also be stored but do not need to be prominently displayed in the UI.

FEAT-1303 - Add Trait Count Roll to Player Generation
----------------------------------------------------------------

Description
Determine how many traits a generated player receives.

Requirements
- During player generation, roll for how many traits the player gets.
- Use this exact distribution:
  - 0 traits = 60%
  - 1 trait = 32%
  - 2 traits = 8%
- This logic must apply to both goalkeepers and outfield players.

Acceptance Criteria
- Generated players receive 0, 1, or 2 traits only.
- The exact probability distribution is implemented.
- Trait count is determined during generation.

----------------------------------------------------------------
FEAT-1304 - Resolve Trait Weighting Context from Player Generation Type
----------------------------------------------------------------

Description
Determine which rarity column should be used when selecting traits for a player.

Requirements
- If the player is a goalkeeper, use the Goalkeeper rarity column.
- If the player is an outfield player, use the rarity column matching the applied influence rule:
  - Influence Defender
  - Influence Defensive Midfielder
  - Influence Attacking Midfielder
  - Influence Attacker
  - Influence Goalscorer
  - No Influence
- Keep this context-resolution logic centralised and reusable.

Acceptance Criteria
- Goalkeepers always use the Goalkeeper rarity column.
- Outfield players use the correct influence-based rarity column.
- No Influence players use the No Influence column.
- The weighting context is resolved correctly for every generated player.

----------------------------------------------------------------
FEAT-1305 - Implement Weighted Trait Selection
----------------------------------------------------------------

Description
Select the actual traits for a player using the resolved rarity weights.

Requirements
- If the player’s trait count roll indicates traits are needed, select traits using the relevant rarity column as weighting.
- Weight values should act as relative rarity/chance weighting.
- A higher rarity value should make that trait more likely to be selected.
- Traits with a weight of 0 in the relevant column must not be selected.
- A player cannot receive the same trait more than once.
- Stop selection once the required number of traits has been assigned.

Acceptance Criteria
- Trait selection is weighted by the correct rarity column.
- Zero-weight traits are never selected for that generation context.
- Duplicate traits are never assigned to the same player.
- Players receive the correct number of selected traits.
