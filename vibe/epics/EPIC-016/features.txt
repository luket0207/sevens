EPIC-016 - Button System Visual Refresh

Overview
The goal of this epic is to refresh the button system across the game so buttons are much more visually distinct, more modern, and more clearly interactive.

At the moment, buttons are blending too much into the surrounding UI because they share too much of the same colour language and general shape as the backgrounds.

This epic will:
- redesign the primary, secondary, and tertiary button styles
- move buttons to their own complementary palette
- improve their shape and borders
- add a stronger modern/stylish visual identity
- add a more impressive hover effect
- preserve button size and position during hover

This epic is about:
- button styling system
- shared reusable button variants
- hover/focus/active visual polish
- improving perceived affordance

This epic is not about:
- changing button behaviour logic
- changing page flows
- changing layout spacing
- adding new button types beyond the current primary/secondary/tertiary set

================================================================
Design Intent
================================================================

Current background palette:
- $primaryBackground: #115752
- $secondaryBackground: #2e7d64
- $tertiaryBackground: #d5ceb5

The updated buttons should use a complementary palette built around:
- teal
- light blue
- blue-teal
- aqua/cyan leaning accents where appropriate

Important visual goal:
- buttons must stand out from the backgrounds
- they must read clearly as buttons
- they should feel modern and stylish
- they should still fit the rest of the game's visual language

================================================================
Core Button Styling Rules
================================================================

Buttons to update:
- Primary
- Secondary
- Tertiary

Shared styling goals:
- stronger visual separation from page backgrounds
- clearer button silhouette
- new shape treatment
- visible border treatment
- more modern finish
- consistent variant hierarchy

Hover rules:
- impressive hover effect
- no scaling on hover
- no movement on hover
- button width/height/position must remain fixed
- hover should feel premium through colour, glow, sheen, border, shadow, inner highlight, or similar visual treatment

Interaction states to consider:
- default
- hover
- active/pressed
- disabled
- focus-visible

Accessibility goal:
- text contrast should remain strong and readable
- the button should remain obviously clickable in all supported states

================================================================
Palette Direction
================================================================

The new button palette should complement the existing background palette without blending into it.

Direction:
- more teal/light-blue than the existing background palette
- visually brighter and clearer than the background surfaces
- hierarchy should still be obvious between primary, secondary, and tertiary

Do not lock the implementation to exactly one hardcoded colour per variant yet if the system benefits from a small token set, but the result should clearly trend toward:
- rich teal-blue for primary
- lighter blue-teal for secondary
- softer pale teal/light neutral for tertiary

The final colours should be chosen to:
- contrast with #115752
- contrast with #2e7d64
- still sit harmoniously beside #d5ceb5

================================================================
Shape and Border Direction
================================================================

Buttons should no longer feel like simple flat blocks that blend into panels.

New shape goals:
- more modern silhouette
- more intentional border radius
- more distinctive outline/border
- premium UI feel

Border goals:
- every button should have a clear border or outline treatment
- border should contribute to the button identity, not just be decorative noise
- border can become more vivid or luminous on hover/focus if appropriate

The styling should feel deliberate and polished rather than generic.

================================================================
Hover Effect Direction
================================================================

Hover effect should feel impressive without changing layout.

Allowed hover ideas include:
- animated gradient shift
- sheen/gloss sweep
- glow or edge-light effect
- stronger border illumination
- inner light/shadow change
- subtle background transition
- layered pseudo-element effects

Not allowed:
- scale change
- translate/movement
- layout shift
- wobble or bounce
- any effect that changes size or position

The hover should make the button feel interactive and premium while remaining stable.

================================================================
System Requirements
================================================================

This should be implemented as a shared button styling system, not one-off per page.

Requirements:
- all primary buttons should share one consistent variant style
- all secondary buttons should share one consistent variant style
- all tertiary buttons should share one consistent variant style
- keep the styling maintainable and reusable
- use shared tokens/mixins/placeholders/utility structure where appropriate for the codebase

The system should be easy to extend later if additional button variants are needed.

FEAT-1611 - Create Bespoke Context-Aware Continue Button for Career Calendar
----------------------------------------------------------------

Description
Replace the current standard button used for the Continue action next to the career calendar with a bespoke Continue button component designed specifically for the calendar flow.

Right now the Continue button only updates its text to reflect what action it will take next.

This feature should keep that behaviour, but expand it so the button is:
- more scannable
- more informative
- more visually tied to the event it is about to trigger
- more distinctive than a normal shared button

This bespoke button should no longer use the normal shared button component.

Requirements
- Create a dedicated Continue button component specifically for the career calendar flow.
- This bespoke component should replace the existing standard button currently used next to the calendar.
- It must still display the action text describing what it will take the user to next.
- The component should be designed for the continue/calendar context only and does not need to behave like a general-purpose button.

Context-aware text
- Keep the existing behaviour where the button text reflects the next action.
- The text should continue to describe what the button will do next, for example:
  - Continue to Match
  - Continue to League Cup Draw
  - Continue to Champions Cup Match
  - Continue to Fixture Generation
  - Continue to Next Day
- Preserve the current logic that determines this text, but move it into or connect it cleanly to the new bespoke component.

Event-based styling
- The button should change its background/style colours depending on the event it will start.
- The colour treatment should align with the existing event colour language already used elsewhere in the calendar system.
- Examples:
  - normal league match -> match yellow
  - Champions Cup match -> purple
  - draw event -> corresponding draw colour
- Use the same event palette logic where practical so the button feels connected to the calendar event system.
- Keep the styling centralised and reusable rather than hardcoding colours in the component body.

Event-based icons
- The button must always show an icon.
- If the next action/event already has an associated icon, use that icon in the button.
- Examples:
  - draw -> box-open
  - cup match -> trophy
  - other event-specific icons already defined in the calendar/event system
- If there is no specific event icon available, show a generic Font Awesome arrow icon.
- The fallback arrow icon should be styled so it feels visually consistent with the rest of the icon system used in the calendar.

Height and layout
- The new Continue button should be 100px tall.
- Keep the width/layout suitable for its existing position next to the calendar.
- The icon and text should be arranged clearly and be easy to scan at a glance.
- The button should feel like a major action panel rather than a small standard button.

Hover behaviour
- The bespoke Continue button should still have a hover effect.
- That hover effect should fit in with the current hover language introduced for the button refresh epic.
- It should not:
  - change scale
  - move position
  - cause layout shift
- It should feel premium and interactive while remaining fixed in size and position.

Design intent
- This button should look more like a contextual action card/button hybrid than a standard UI button.
- The user should be able to understand:
  - what happens next
  - what type of event it is
  - how important it is
with a quick glance.

Implementation notes
- Do not reuse the generic button component for this one.
- Create a dedicated component and dedicated styling for this Continue button.
- Reuse existing calendar event metadata where sensible:
  - icon mappings
  - colour mappings
  - action label logic
- Keep the component integrated with the calendar progression flow.

Acceptance Criteria
- The Continue button next to the calendar is replaced with a bespoke component.
- The bespoke Continue button no longer uses the shared generic button component.
- It still displays the correct context-aware action text.
- The button always displays an icon.
- If the next action has an event icon, that icon is shown.
- If no event icon exists, a generic Font Awesome arrow is shown.
- The button changes visual style/colour depending on the event it will trigger.
- The button is 100px tall.
- It has a hover effect that matches the current premium button hover language.
- The hover effect does not change the button’s size or position.
- The result is more scannable, more informative, and more visually tied to the next calendar action.
