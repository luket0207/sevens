EPIC-002 - Feature Prompt File

This epic fills in the remaining gaps from the career team setup flow created in the previous epic.

The main goals of this implementation pass are:
- Build the kit selection component for the player's team.
- Save the kit data into the team data in game state.
- Build a reusable shirt render/decoder component that can reproduce the selected shirt visually from saved values.
- Add goalkeeper kit selection.
- Add the first mechanics for team/player generation through a temporary debug player generator tool on the career start page.
- Generate players with names, appearance values, skill ratings, and a calculated overall rating.

Scope for this prompt file:
- Implement the team kit selection flow.
- Implement shirt data modelling and shirt rendering/decoding.
- Implement goalkeeper kit selection.
- Add a temporary player generator debug tool in place of the real team selector.
- Generate a single random player from selected overall rating and position.
- Display all generated player details including raw skills and calculated overall.

Important scope rules:
- Do not build the full team selector yet.
- Do not build the full squad builder yet.
- Do not treat the temporary player generator as final game UI.
- The debug player generator is only a temporary interface to exercise the underlying generation mechanics.
- The underlying player generation logic should still be written in a reusable way because it will be used later by the real game systems.
- The layout should remain full-width and responsive.
- Keep the code modular and aligned to the existing project patterns.

FEAT-001 - Add team kit data structure to career team setup
================================================================

Purpose:
Expand the team setup state so it can store the full kit information needed by the game.

Outcome:
The player's team can now store home kit, away kit, home colour, away colour, and goalkeeper kit data in a structured way.

Prompt:
Implement FEAT-001 for EPIC-002.

Goal:
Add the required kit data structure to the player team setup state and prepare it to be saved into the team data in gameContext.

Requirements:
- Extend the team setup/team data shape so it can store:
  - homeKit
  - awayKit
  - homeColour
  - awayColour
  - goalkeeperKit
- The shirt data model should support:
  - a main colour
  - an optional detail colour
  - a shirt pattern/type
- The main colour of the home shirt becomes homeColour.
- The main colour of the away shirt becomes awayColour.
- Goalkeeper kit is a single selection shared across both home and away goalkeeper use.

Implementation notes:
- Keep the data shape clean and serialisable.
- The shirt value should be something that can later be decoded by a renderer component to recreate the exact shirt.
- Prefer a simple structured object rather than a hard-to-maintain encoded string unless the existing architecture strongly benefits from encoding.
- This feature is about state shape and persistence readiness, not the full editing UI.

Acceptance criteria:
- Team/team setup state now includes homeKit, awayKit, homeColour, awayColour, and goalkeeperKit.
- homeColour and awayColour derive from the main shirt colours.
- The data shape is reusable and suitable for saving in gameContext.

FEAT-004 - Enforce home vs away colour contrast rules
================================================================

Purpose:
Prevent kit clashes by ensuring the away kit contrasts clearly with the home kit.

Outcome:
The away shirt main colour must maintain sufficient contrast from the home shirt main colour, including when randomised.

Prompt:
Implement FEAT-004 for EPIC-002.

Goal:
Add contrast validation and enforcement between home and away shirt main colours.

Requirements:
- The away shirt main colour must have good visual contrast with the home shirt main colour.
- This rule must apply both when the player selects colours manually and when colours are randomised.
- Prevent or correct invalid away colours that are too close to the home colour.
- Surface the rule clearly in the UI so the player understands why some combinations are blocked or adjusted.

Implementation notes:
- Use a simple practical contrast rule suitable for game kit clash avoidance.
- The exact implementation can be colour-distance based, luminance based, or another straightforward method, as long as it is deterministic and produces clearly different main shirt colours.
- Prefer a solution that is robust and easy to reason about rather than visually perfect.
- Make the logic reusable for future random generation.

Acceptance criteria:
- Home and away main colours cannot end up too similar.
- Manual selection respects the contrast rule.
- Randomisation respects the contrast rule.
- The logic is reusable and modular.
- The saved away colour always has acceptable contrast with the home colour.
