EPIC-005 - Career Generation, League Setup, and Team Creation

Overview
The goal of this epic is to generate the full starting career data after the user clicks "Start Career" on /career/start.

This epic introduces:
- a career generation loading screen
- generation of all leagues and AI teams
- generation of all team data
- generation of all players for every team
- saving the generated career structure into gameContext
- a debug view on /career/home to inspect the generated output

This is the first real career-generation epic. The focus is on producing the full starting competition structure and team data before the player enters the main career home screen.

================================================================
Core Career Generation Scope
================================================================

When the user clicks "Start Career" on /career/start:

1. The app should leave /career/start
2. A loading screen should be shown while the career is being generated
3. The system should generate all required leagues and teams
4. Each team should be generated with all required data
5. All generated career data should be saved into gameContext
6. Once complete, the app should navigate to /career/home
7. /career/home should expose a debugging view to inspect the generated leagues, teams, and players

================================================================
Competition Structure
================================================================

The career must generate the following teams and league groupings:

- 7 teams rated 20 to 25 - League 5
- 8 teams rated 25 to 30 - League 4
- 8 teams rated 30 to 35 - League 3
- 8 teams rated 35 to 40 - League 2
- 8 teams rated 40 to 45 - League 1
- 7 teams rated 40 to 50 - Foreign teams for the Champions Cup

This gives a total of:
- 39 domestic league teams
- 7 foreign teams
- 46 generated AI teams overall

These should be generated fresh when a new career is started.

================================================================
Player Generation Model for Teams
================================================================

Each generated team must contain exactly 7 players:
- 1 goalkeeper
- 2 defenders
- 2 midfielders
- 2 attackers

The team overall should be used to derive the 7 player overall targets.

Rule:
- create 7 player overall target values
- those 7 values must average exactly to the team's overall value
- they should have a spread of up to 5 either side of the team overall where possible

Example:
If team overall is 25, valid player overall targets could look like:
[20, 27, 28, 24, 25, 26, 25]

Then assign them as:

1. first value -> goalkeeper
2. second value -> defender 1
3. third value -> defender 2
4. fourth value -> midfielder 1
5. fifth value -> midfielder 2
6. sixth value -> attacker 1
7. seventh value -> attacker 2

Generation rules by slot:

Goalkeeper
- generate as goalkeeper using the goalkeeper player type

Defenders
- both defenders must be generated as outfield players
- both defenders must force:
  - Influence Defender

Midfielders
- both midfielders must be generated as outfield players
- each midfielder must force either:
  - Influence Defensive Mid
  - Influence Attacking Mid
- each midfielder should choose between the two with a 50/50 chance

Attackers
- both attackers must be generated as outfield players
- each attacker must force either:
  - Influence Attacker
  - Influence Goalscorer
- each attacker should choose between the two with a 50/50 chance

================================================================
Generation Efficiency Expectations
================================================================

This epic should aim for a good balance between:
- believable randomness
- maintainable logic
- acceptable generation speed

The user is comfortable with career generation taking around 20 to 30 seconds if needed, so absolute optimisation is not required at the expense of variety.

However, the implementation should still make sensible attempts to avoid wasteful generation.

Guidance:
- Reuse shared generators and utilities rather than duplicating logic
- Avoid unnecessary repeated setup work inside loops
- Prefer clean batched generation where practical
- Team names, stadium names, and similar data can be generated from curated pools/templates rather than expensive bespoke logic every time
- Player skill generation should remain properly randomised through the shared player generator
- If useful, use a simple prebuilt pool/schema approach for team names, stadium names, and similar decorative data
- Choose the approach that best balances strong random-feeling output with efficient implementation

The purpose of this epic is not micro-optimisation, but it should still be implemented sensibly.

================================================================
Loading Screen Behaviour
================================================================

A loading screen is required for career generation.

This loading screen should appear after the user clicks "Start Career" on /career/start and before they arrive on /career/home.

Requirements:
- show a loading spinner or equivalent loading visual
- show progress feedback
- progress feedback should try to be meaningfully accurate
- the screen should communicate that the career is being generated

The progress system should reflect actual generation work as much as reasonably possible.

Examples of useful progress stages:
- preparing career data
- generating leagues
- generating teams
- generating players
- finalising career
- saving to gameContext

The exact UI wording can vary, but the user should feel the process is genuinely progressing, not frozen.

================================================================
Career Home Debug View
================================================================

Once generation is complete and the user lands on /career/home, the generated data should be inspectable through a debug view.

Requirements:
- display the resulting generated array/data structure for leagues and teams
- include full team details
- include player details
- use an expandable nested accordion-style arrangement
- make it easy to scan:
  - leagues
  - teams inside leagues
  - players inside teams

This is a debug/development aid so the generated structure can be verified clearly.

- Defenders always use Influence Defender.
